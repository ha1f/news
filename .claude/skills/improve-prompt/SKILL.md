# プロンプト改善スキル

`$ARGUMENTS` で指定されたプロンプトを、Anthropicの公式ベストプラクティスに基づいて分析・改善する。

## 原則

Anthropic公式ドキュメント（https://platform.claude.com/docs/en/docs/build-with-claude/prompt-engineering/claude-4-best-practices）に基づく。

### 守るべき原則

1. **明確で直接的に書く**: Claudeは明示的な指示に最もよく応答する。曖昧な表現を避け、具体的に何を求めるか書く
2. **理由を添える**: なぜそうすべきかの動機を伝えると、Claudeはそこから汎化できる。ルールだけでなく背景を書く
3. **構造化する**: 指示・コンテキスト・入力をXMLタグで分離し、誤解を減らす
4. **肯定文で指示する**: 「〜しないで」より「〜する」の形で書く

### 避けるべきパターン

1. **具体例の過剰使用**: 例はプロンプトを長くし、本質を薄める。原則を伝えればClaudeは汎化できる
2. **過剰な強調**: 以前のモデルで必要だった「CRITICAL」「MUST」等の強い言葉は、最新モデルでは過剰反応を引き起こす。通常の言葉で十分
3. **冗長な指示**: 同じことを別の言い方で繰り返さない。1回で明確に伝える
4. **過度な防御的指示**: 起こりえないシナリオへの対処を書かない。実際の課題にフォーカスする

## ワークフロー

### Phase 1: 現状分析

対象プロンプトを読み、以下を特定する:

- プロンプトの目的と対象モデル
- 現在の構造（フラットテキスト / XMLタグ / セクション分け）
- 冗長な箇所、重複、不要な例示
- 「なぜ」が欠けている指示
- 否定形で書かれた指示
- 過剰な強調表現

### Phase 2: 改善案の作成

分析結果に基づき、改善版を作成する。改善時のルール:

- **削る勇気**: 不要な例・冗長な説明・重複する指示を削除する
- **理由を追加**: ルールだけの指示に「なぜそうすべきか」を一文追加する
- **肯定文に変換**: 否定形の指示を肯定形に書き換える
- **強調を下げる**: 「MUST」「CRITICAL」「NEVER」等を通常の表現に置き換える
- **構造を整理**: 長いプロンプトはXMLタグやセクションで整理する

### Phase 3: 差分の提示

改善前後の差分を提示し、各変更の理由を説明する。ユーザーが判断できるよう、変更ごとに「何を」「なぜ」変えたかを簡潔に述べる。
