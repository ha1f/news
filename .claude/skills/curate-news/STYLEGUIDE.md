# スタイルガイド

このスキルを構成するファイルの書き方・設計方針。

## 全体方針

- **関心の分離**: SKILL.md はワークフロー（何をどの順でやるか）、sources はデータ仕様（どこから何をどう取るか）
- **ソース非依存**: SKILL.md に特定ソース固有の情報（URL、フィード形式、名前空間等）を書かない
- **最新仕様への柔軟性**: 具体的な実装コードを書きすぎない。仕様とフィールドマッピングを書き、実装はモデルに任せる
- **厳密さよりも柔軟さ**: このガイド自体も厳密に守る必要はない。LLMが読んで意図を汲み取れれば十分。フォーマットも参考程度でよい

## SKILL.md

- ワークフローの手順・判断基準・出力フォーマットを定義する
- ソース定義の参照方法（パス、Glob パターン）は書くが、中身の詳細は書かない
- キャッシュの構造やTTL判定など、ソース横断の共通仕様は書いてよい
- 取得時の原則（並列取得、エラーハンドリング等）は抽象的なルールとして書く

## sources/*.md

各ソース定義ファイルの構成:

1. **メタ情報**: ソースID、TTL、説明
2. **カテゴリとフィードURL**: テーブル形式（カテゴリ、URL、含まれるトピック）
3. **仕様リンク**: 公式APIドキュメントやフィード仕様へのURL
4. **フィード形式と取得上の注意**: フィードの種類（RSS 1.0/2.0、JSON API等）と、取得時に注意すべき点（認証、レート制限、2段階取得の要否など）
5. **フィールドマッピング**: ソースのフィールド → キャッシュの `items` フィールドへの対応表
6. **表示名**: 出力でソース名として使う短縮名

### フォーマット例

```markdown
# {ソース名}

- **ソースID**: {英数字のID}
- **TTL**: {N}分
- **説明**: {ソースの概要を1行で}

## 仕様リンク

- {公式APIドキュメントやフィード仕様のURL}

## カテゴリとフィードURL

| カテゴリ | フィードURL | 含まれるトピック |
|---------|-----------|----------------|
| {カテゴリ名} | `{URL}` | {トピック1}、{トピック2}、... |

## フィード形式と取得上の注意

- {フィード形式（RSS 1.0 / RSS 2.0 / JSON API 等）}
- {取得時の注意点（認証、レート制限、2段階取得、名前空間等）}

## フィールドマッピング

| ソースのフィールド | キャッシュのフィールド | 備考 |
|---|---|---|
| {フィールド名} | title | |
| {フィールド名} | url | {補足があれば} |
| {フィールド名} | published_at | {日付形式の補足} |
| {フィールド名} | meta.{キー} | {型変換等の補足} |

## 表示名

`{短縮名}`
```

### 書かないこと

- **完全な実装コード**: python3のパーサー全体など。フィールドマッピングと注意点があればモデルが実装できる
- **他ソースとの比較や共通処理**: それはSKILL.mdの責務

## preferences.md

- **興味・関心**: キュレーション対象のトピック。ソースのカテゴリとマッチングして自動選択に使う
- **品質基準**: どういう記事を選びたい/避けたいか。キュレーション時のフィルタリングに使う
- ソース固有の設定（有効/無効、カテゴリ指定等）は書かない — マッチングで自動選択する

## cache/

- キャッシュの構造（JSON スキーマ）は SKILL.md に定義
- ファイル名規則: `{ソースID}-{カテゴリ}.json`（`/` は `-` に置換）
- gitignore 対象

## output/

- キュレーション結果の markdown を日付別に保存（`{YYYY-MM-DD}.md`）
- gitignore 対象
